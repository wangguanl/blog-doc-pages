调整、改动：需要先知道之前是如何运作的， 了解之后 改变不是靠一个人来做的，想做这个事情堆积到前端一个人身上：调研阻力，为了图个人省事，不支持改动， 我挺积极的去做事情， 但是四处碰壁

拿到经过多代“创作者”之手的代码，虽然前人付出了许多心血，但是到了我们手上怎么看都没有原创的极致。注释、文档大部分是没有的。即使有些残留的珍贵文档信息，也已经和代码对不上，时代感鲜明。想根据现有的代码了解之前的思路和意图，可能全靠猜。

目前工作流程：上下游没有打通，产品主导变成了前端主导 时间圆盘分析：10%对需求、40%开发、30%和产品扯皮改需求（在开发阶段在寻找不合理的东西）、10%对接后端、10%对接测试 技术问题在技术之外花费的功夫更大。

我能做 + 我想做 + 需要我 = 核心价值 我能做 + 我想做 + 不需要我 = 潜在价值 我能做 + 我不想做 + 需要我 = 例行工作 我能做 + 我不想做 + 不需要我 = backup 我不能做 + 我想做 + 需要我 = 成长空间 我不能做 + 我想做 + 不需要我 = 自我追求 我不能做 + 我不想做 + 需要我 = 无法胜任 我不能做 + 我不想做 + 不需要我 = 无需关注

不同意。领导只会让你搞好，不会指定你提的方案，只会说我不懂，你把它弄好。要知道，能当上领导的不但没有傻子，恰恰相反都精着呢。

1.让公司的技术负责人以及尽可能多的领导知道目前的状况。 2.要让他们清楚重构和彻底重写分别面临哪些困难，如果失败会面临什么境况。 3.你只给出意见，一定要让领导做出决定。留下可以追溯的证据，例如邮件或会议记录。
